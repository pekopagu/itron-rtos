# 修正計画: v1.0 Claude Codeレビュー対応

## 1. レビュー指摘の再分類

### Critical

- C-1: `dispatcher_switch_to()` が `task_mark_running()` / `task_mark_ready_from_running()` を使わず、TCB の `state` を直接書き換えている。
- C-2: `wai_sem()` / `slp_tsk()` などで WAITING に入るタスクの実行コンテキストが保存されず、wakeup 後に待ち呼び出し元へ復帰できない。
- C-3: task context API に割り込み保護がなく、task context 由来の dispatch pending が timer IRQ 側処理と競合する。

### Major

- M-1: `task_register()` が `task_configure_slot()` を使わず、登録初期化ロジックが重複している。
- M-2: 数値出力ヘルパーと状態文字列ヘルパーが複数ファイルに重複している。
- M-3: `sem_legacy_sig_sem_disabled()` が未使用のまま公開ヘッダに残っている。
- M-4: `task_mark_waiting_on_sem()` が READY から WAITING への遷移を許可している。
- M-5: `delay_queue.c` が `task->delay_ticks_remaining` を直接書き換えている。
- M-6: `preemption_evaluate_from_irq()` が IRQ 評価開始時に task context 由来の `dispatch_pending` まで消去し得る。

### Minor

- m-1: `task_dump()` の Doxygen コメント位置が実体とずれている。
- m-2: `TASK_ID_MAX` / `SEM_ID_MAX` が `INT_MAX` 相当のマジックナンバーになっている。
- m-3: `scheduler_select_next()` が毎回 `MAX_TASKS` 全スロットを線形走査している。
- m-4: `sem_reset_wait_queue()` が初期化時に `MAX_TASKS` 分を常にゼロクリアしている。
- m-5: エラーコード値が μITRON 4.0 仕様値と異なる。

### Suggestion

- S-1: `dispatch_pending` を複数要求対応へ拡張する。
- S-2: ログ出力を一元化し、将来の ringbuffer 化へ備える。
- S-3: `task_context.c` の smoke 用グローバル変数を分離する。
- S-4: `rel_wai()` / `sus_tsk()` に向けて `wait_reason` と起床 API を拡張する。

## 2. 今回修正する対象

最大3件として、次を選ぶ。

1. C-1: dispatcher による TCB state 直接書き換えの排除
2. C-3 / M-6: task context 由来 dispatch pending を IRQ 開始時に失わないようにする
3. M-4: `task_mark_waiting_on_sem()` を RUNNING からの遷移に限定する

選定理由は、いずれも状態遷移、dispatcher の責務、IRQ 安全性に直結し、API 名を大きく変えずに局所的な修正として扱えるためである。C-2 は最重要だが、実行コンテキスト保存/復元の完成が必要で変更範囲が大きいため、今回の小さな修正セットからは外す。

## 3. 修正方針

### 3.1 C-1: dispatcher による TCB state 直接書き換えの排除

* 指摘内容
  * `kernel/dispatcher.c` の `dispatcher_switch_to()` が `from->state = TASK_STATE_READY` と `to->state = TASK_STATE_RUNNING` を直接実行している。
  * task module が持つ状態遷移 API を迂回するため、状態所有権、ログ、待ちメタデータのクリアが不整合になる。
* 対象ファイル
  * `kernel/dispatcher.c`
  * 必要に応じて `kernel/include/dispatcher.h`
* 対象関数
  * `dispatcher_switch_to()`
  * 既存 API として `task_mark_ready_from_running()`
  * 既存 API として `task_mark_running()`
* 修正方針
  * `from` が RUNNING の場合は、直接 `state` を書き換えず `task_mark_ready_from_running(from->id)` を呼ぶ。
  * `from` が READY の場合は、`yield_tsk()` が先に READY 化している既存経路として維持し、再度 `task_mark_ready_from_running()` は呼ばない。
  * `from` が WAITING の場合は、待ち入り済みタスクなので READY 化しない。
  * `to` の RUNNING 化は `task_mark_running(to->id)` で行い、成功した後だけ `current_task = to` を更新する。
  * 各 API 呼び出しに失敗した場合は `current_task` を更新せず、既存の `DISPATCHER_ERR_*` へ変換して返す。
* 影響範囲
  * 状態遷移への影響: RUNNING->READY と READY->RUNNING の所有権が task module に戻る。RUNNING 化時に `wait_reason` / `wait_sem_id` / `delay_ticks_remaining` のクリアが保証される。
  * IRQ安全性への影響: 直接の IRQ 保護は追加しない。ただし dispatcher が状態を部分更新したまま `current_task` だけ進める危険を減らす。
  * 既存APIへの影響: public API 名は変更しない。`dispatcher_switch_to()` の戻り値は既存のエラー体系に合わせる。
* 動作確認方法
  * `make` でビルドする。
  * `make run` で QEMU smoke を実行し、`docs/logs/qemu-serial.log` に既存の dispatcher/context/yield/semaphore/delay/sleep smoke が出ることを確認する。
  * `dispatcher_switch_to()` 経由の RUNNING 化後に、task dump/log で古い wait メタデータが残らないことを確認する。
* リスク
  * `task_mark_running()` は READY のみ許可するため、`to` が別経路で既に RUNNING になっている smoke があれば失敗する。
  * `from` が READY の yield 経路と WAITING の wait 経路を誤って READY 化すると、既存 smoke の前提を壊す。

### 3.2 C-3 / M-6: task context 由来 dispatch pending を IRQ 開始時に失わない

* 指摘内容
  * `preemption_evaluate_from_irq()` が先頭で `dispatch_pending_clear_for_test_or_later_boundary()` を呼び、`sta_tsk()` / `sig_sem()` など task context 由来の pending を消去し得る。
  * task context API 全体には割り込み保護がなく、timer IRQ と dispatch pending 更新が競合する。
* 対象ファイル
  * `kernel/preemption.c`
  * `kernel/dispatch_pending.c`
  * `kernel/include/dispatch_pending.h`
  * 参照確認として `kernel/itron_api.c`
* 対象関数
  * `preemption_evaluate_from_irq()`
  * `dispatch_pending_clear_for_test_or_later_boundary()`
  * `dispatch_request_from_irq()`
  * `dispatch_request_from_task_wakeup()` など task context 由来の pending 設定関数
* 修正方針
  * 大規模な pending queue は導入しない。
  * `dispatch_pending` に由来を区別できる最小の情報を持たせる。候補は `DISPATCH_PENDING_SOURCE_IRQ` と `DISPATCH_PENDING_SOURCE_TASK` のような内部 enum である。
  * IRQ 評価開始時の clear は、IRQ 由来 pending だけを対象にする。task context 由来 pending が残っている場合は消さない。
  * 既存の smoke 用全 clear API は名前を変えずに残し、boot-time smoke の明示的な初期化用途では従来どおり全消去できるようにする。
  * `preemption_evaluate_from_irq()` は「IRQ 評価用 clear」へ置き換える。全 clear は呼ばない。
* 影響範囲
  * 状態遷移への影響: task state そのものは変更しない。READY 化済みタスクへの dispatch 要求が IRQ によって消える不整合を減らす。
  * IRQ安全性への影響: pending 消失の主原因を局所的に潰す。ただし `cli`/`sti` ベースの完全なクリティカルセクションは今回は導入しないため、C-3 全体は完了ではなく段階対応とする。
  * 既存APIへの影響: public な ITRON API 名は変更しない。`dispatch_pending` の内部構造と補助 API は変更される可能性がある。
* 動作確認方法
  * `make` でビルドする。
  * `make run` で `preempt-irq`、`dispatch-pending`、`create-start-smoke`、`sleep-wakeup-smoke` のログを確認する。
  * `sta_tsk()` / `sig_sem()` 系 smoke の直後に timer IRQ 評価が走っても、task context 由来 pending が予期せず `no-pending` にならないことをログで確認する。
* リスク
  * pending の source 分離を誤ると、古い IRQ pending が残り続けて不要な dispatch を誘発する。
  * 全 clear を残す箇所と IRQ 用 clear を使う箇所の切り分けを誤ると、既存 smoke のログ順序が変わる。
  * 割り込み禁止による完全保護ではないため、C-3 の根本対策は別途必要である。

### 3.3 M-4: `task_mark_waiting_on_sem()` を RUNNING からの遷移に限定する

* 指摘内容
  * `task_mark_waiting_on_sem()` が `TASK_STATE_READY` と `TASK_STATE_RUNNING` の両方から WAITING への遷移を許可している。
  * `wai_sem()` は task 文脈 API なので、WAITING に落ちる対象は current RUNNING task に限定すべきである。
* 対象ファイル
  * `kernel/task.c`
  * `kernel/include/task.h`
  * smoke 調整が必要な場合のみ `kernel/kernel.c`
* 対象関数
  * `task_mark_waiting_on_sem()`
  * 参照元として `wai_sem()`
  * smoke 参照元として `kernel_run_sleep_wakeup_task_api_smoke()` など
* 修正方針
  * `task_mark_waiting_on_sem()` の許可条件を `TASK_STATE_RUNNING` のみに変更する。
  * Doxygen コメントも「観測用に READY を許す」説明から、task 文脈 API として RUNNING のみ許可する説明へ更新する。
  * READY task を直接 WAITING に落としている smoke がある場合は、先に `dispatcher_commit_current()` または既存の current 確定経路で RUNNING 化してから呼ぶ形に調整する。
  * `task_mark_waiting_on_delay()` と `task_mark_waiting_on_sem_timeout()` は既に RUNNING 限定なので、仕様を揃える。
* 影響範囲
  * 状態遷移への影響: READY->WAITING の不正遷移を拒否し、RUNNING->WAITING のみを正規経路にする。
  * IRQ安全性への影響: 直接の IRQ 保護は追加しない。ただし current でない READY task が WAITING 化されることで scheduler/dispatcher と pending が食い違うリスクを減らす。
  * 既存APIへの影響: 関数名と引数は変更しない。READY task に対して呼んだ場合の戻り値が成功から `TASK_ERR_BAD_STATE` へ変わる。
* 動作確認方法
  * `make` でビルドする。
  * `make run` で `sem-smoke`、`sleep-wakeup-smoke`、`tick-ready-smoke` を確認する。
  * READY task に対する `task_mark_waiting_on_sem()` の invalid-state 観測が必要なら、既存 smoke の期待ログを `TASK_ERR_BAD_STATE` 前提に揃える。
* リスク
  * 古い章の smoke が READY から直接 WAITING を作る前提を持っている場合、実装修正だけではログが変わる。
  * `sem_enqueue_waiter()` と `task_mark_waiting_on_sem()` の呼び順を誤ると、semaphore wait queue にだけ残るタスクが発生する。

## 4. 今回は見送る項目

- C-2: ブロッキング待ちからの復帰コンテキスト保存なし
  - 最重要だが、`arch_context_switch`、`task_context_t`、WAITING 入口、wakeup 復帰、interrupt return boundary にまたがる設計変更になる。今回は大規模リファクタリングを避け、C-1 と M-4 で状態所有権を先に整える。
- M-1: `task_register()` が `task_configure_slot()` を使っていない
  - 局所的で有効な保守性改善だが、今回の優先観点である不正状態遷移と IRQ 安全性より優先度を下げる。
- M-2: 数値出力ヘルパー重複
  - 9ファイル横断の整理になり、ログ順序や教育用の章立てに影響しやすい。今回の品質改善では対象外にする。
- M-3: `sem_legacy_sig_sem_disabled()` の削除
  - デッドコード削除として妥当だが、RTOS の状態遷移や IRQ 安全性には直接効かない。
- M-5: `delay_queue.c` による `delay_ticks_remaining` 直接書き換え
  - 所有権上は問題だが、今回の3件に比べると状態破壊の即時性が低い。M-4 の後に取り組むと、WAITING モデルを壊しにくい。
- Minor 全件
  - 今回のルールにより修正対象候補から外す。
- Suggestion 全件
  - 将来設計案であり、今回の修正対象候補から外す。

## 5. 実装時の注意点

- scheduler は READY task の選択に留め、task state や `current_task` を更新しない。
- dispatcher は切替境界と `current_task` 更新を担当するが、TCB state の直接書き換えは task module API 経由に寄せる。
- `current_task` は、切替先の RUNNING 化が成功した後にだけ更新する。
- WAITING task を READY に戻す処理では、待ち理由に応じて semaphore wait queue と delay queue の整合性を崩さない。
- `wai_sem()` の待ち入りは RUNNING current task のみを対象にする。READY task を直接 WAITING に落とさない。
- task context 由来の dispatch pending と IRQ 由来の dispatch pending を混同しない。
- IRQ handler 内で直接 `dispatcher_switch_to()` へ進まない既存方針を維持する。
- 今回は Mutex / EventFlag / Mailbox / MemoryPool を実装しない。
- public な ITRON 風 API 名は大きく変更しない。
- C-2 の完全解決までは、blocking wait からの本格的な再開が未完成であることをログとコメントで過剰に隠さない。

## 6. 次にCodexへ渡す実装修正指示案

docs/reviews/v1.0/v1.0-codex-fix-plan.md に従って、μITRON風RTOS v1.0 の Claude Codeレビュー対応を実装してください。

今回の修正対象は次の3件に限定します。

1. C-1: `kernel/dispatcher.c` の `dispatcher_switch_to()` から TCB state の直接書き換えをなくし、`task_mark_ready_from_running()` と `task_mark_running()` を使って状態遷移してください。`current_task` は `task_mark_running(to->id)` 成功後にだけ更新してください。
2. C-3 / M-6: `preemption_evaluate_from_irq()` が task context 由来の dispatch pending を消さないよう、`dispatch_pending` に最小限の由来区別を追加してください。IRQ 評価開始時は IRQ 由来 pending のみを clear し、既存 smoke 用の全 clear API は必要箇所でだけ使ってください。
3. M-4: `task_mark_waiting_on_sem()` を RUNNING task からの遷移に限定し、READY からの WAITING 遷移を `TASK_ERR_BAD_STATE` にしてください。関連する Doxygen コメントと必要最小限の smoke 調整も行ってください。

禁止事項:

- C-2 のコンテキスト保存/復元完成には着手しない。
- Mutex / EventFlag / Mailbox / MemoryPool は実装しない。
- API 名を大きく変更しない。
- ログ共通化や ready queue 化など、今回選んでいない指摘は実装しない。
- 既存RTOSコードを参照・コピーしない。

実装後は `make` と `make run` を実行し、`docs/logs/qemu-serial.log` で dispatcher、preempt-irq、dispatch-pending、sem-smoke、sleep-wakeup-smoke、tick-ready-smoke の退行がないことを確認してください。
