# μITRON風RTOS v1.0 アーキテクチャレビュー

**レビュー実施日:** 2026-06-12  
**対象バージョン:** v1.0  
**レビュアー:** Claude Sonnet 4.6 (シニアRTOSエンジニア兼ソフトウェアアーキテクト視点)  
**対象ブランチ:** main

---

## Executive Summary

教育目的の μITRON 風 RTOS として、**全体的な設計品質は高い**。特に以下の点を高く評価する。

- **HAL 境界の一貫性**: `kernel → HAL → arch(x86_64) → serial` の依存方向が全ファイルで守られている
- **状態所有権の明確化**: TCB 状態の変更は task モジュール API へ集約する方針が確立されている
- **段階的実装とログ観測モデル**: 各章で「動くものを小さく作り観測する」という教育的アプローチが一貫している
- **Doxygen コメント**: 主要な public API には詳細な意図と制約の記述がある

一方、**v2.0 に向けて解決すべき重大な設計問題**が存在する。特に「ブロッキング待ちからの復帰」メカニズムが架空の観測モデルに留まり、実際のタスク再開ができない点は、RTOS としての根幹に関わる。また dispatcher による TCB 直接書き換え、IRQ 保護の欠如など、並行性に関する問題も v2.0 前に整理が必要である。

---

## Critical — 重大な設計問題

### C-1: dispatcher が task module の状態遷移 API を迂回している

**ファイル:** `kernel/dispatcher.c:264-275`

```c
// dispatcher_switch_to() 内
if (from->state == TASK_STATE_RUNNING) {
    from->state = TASK_STATE_READY;   // ← TCB 直接書き換え
}
to->state = TASK_STATE_RUNNING;       // ← TCB 直接書き換え
current_task = to;
```

**問題:** task モジュールは `task_mark_running(id)` および `task_mark_ready_from_running(id)` という正式な状態遷移 API を提供している。しかし `dispatcher_switch_to()` はこれらを使わず、`tcb_t *` の mutable ポインタ経由で直接 `state` フィールドを書き換えている。

これにより次の問題が生じる:

1. **task モジュールのログが発火しない**: `task_mark_running()` が持つ状態遷移ログ (`[task] waiting: ...` 等) が dispatcher 経由の切り替えでは出力されない
2. **state 所有権の二重性**: TCB state は task モジュールが所有するという原則が破れ、将来の不変条件維持が困難になる
3. **wait_reason / wait_sem_id のクリアが保証されない**: `task_mark_running()` は RUNNING 遷移時に wait メタデータをクリアするが、直接書き換えではそれが行われない

**修正方針:** `dispatcher_switch_to()` 内の直接書き換えを `task_mark_running(to->id)` および `task_mark_ready_from_running(from->id)` 呼び出しに置き換える。

---

### C-2: ブロッキング待ちからの復帰コンテキストが保存されていない

**ファイル:** `kernel/dispatcher.c:277-291`, `kernel/task_context.c:357-388`

`wai_sem()` や `slp_tsk()` でタスクが WAITING に落ちると、`dispatcher_switch_to()` は次のパスを通る:

```c
// dispatcher_switch_to() 内
if (from->state == TASK_STATE_WAITING) {
    result = task_context_prepare_initial_frame(to);  // to に初期フレームを準備
    if (result == DISPATCHER_OK) {
        result = task_context_switch_to_task(to);     // boot_context から to へ切り替え
    }
    return result;
}
```

`task_context_switch_to_task(to)` は `arch_context_switch(&boot_context, &to->context)` を呼ぶ。つまり **WAITING になった from タスクのコンテキストは保存されない**。

**影響:** WAITING タスクが wakeup されて再度 dispatch された場合、`context.rsp` は登録時の `stack_top` を指す初期状態のままである。`task_context_prepare_initial_frame()` が再度呼ばれるか、または古い初期フレームへ ret するため、**タスクは待ち呼び出しの戻り先ではなく entry() の先頭から再実行される**。

v1.0 の boot-time smoke では「1回しか実行しない」前提のため問題が顕在化していないが、現設計では本格的な blocked/resume サイクルを実現できない。

**修正方針:** WAITING になる前にタスクの実行コンテキスト (rsp, rbp, rbx, r12-r15, rip) を TCB に保存し、wakeup 後に復元するメカニズムが必要。これは v2.0 の最優先実装課題である。

---

### C-3: task context API に割り込み保護がない

**ファイル:** `kernel/itron_api.c` (全 API 関数), `kernel/preemption.c:213`

`wai_sem()`, `sig_sem()`, `dly_tsk()`, `twai_sem()`, `slp_tsk()`, `wup_tsk()`, `sta_tsk()` はいずれも `cli`/`sti` や相当するクリティカルセクション保護を持たない。

**具体的なレース条件:**

```
[task context]  sig_sem() 呼び出し
                  → sem_dequeue_waiter() でタスクを READY 化
                  → dispatch_request_from_task_wakeup() で pending を set

[timer IRQ]     割り込み発生 (ここで preempt される)
                  → preemption_evaluate_from_irq() 内で
                  → dispatch_pending_clear_for_test_or_later_boundary() を呼ぶ
                  → task context が設定した pending が消去される
```

`preemption.c:213` で `dispatch_pending_clear_for_test_or_later_boundary()` が毎 IRQ 先頭に呼ばれているため、task context 由来の pending が IRQ に上書きされる。

**修正方針:** `cli`/`sti` ベースの spinlock 相当の保護、または dispatch_pending の複数 reason 対応 (task-context 由来 pending を IRQ がクリアしない設計) が必要。

## Major — 優先的に修正したい問題

### M-1: `task_register()` が `task_configure_slot()` を使っていない

**ファイル:** `kernel/task.c:462-534` (task_register), `kernel/task.c:310-351` (task_configure_slot)

第14章で `task_configure_slot()` が追加されたが、既存の `task_register()` はリファクタリングされていない。`task_register()` の lines 490-502 は `task_configure_slot()` と本質的に同一のロジックを持つ重複コードになっている。

```c
// task_register() 内 (l.490-502) — task_configure_slot() と同等
slot->id = id;
slot->name = name;
slot->entry = entry;
slot->priority = priority;
slot->state = TASK_STATE_READY;
slot->wait_sem_id = 0;
// ...
```

**修正方針:** `task_register()` を `task_configure_slot(slot, id, name, entry, priority, stack_base, stack_size, TASK_STATE_READY)` に置き換える。

---

### M-2: 数値出力ヘルパーが9ファイルに重複実装されている

`write_uint` / `write_int` / `write_hex` の本質的に同一な実装が以下のファイルに存在する:

| ファイル | 関数名 |
|---------|--------|
| `kernel/task.c` | `task_write_uint`, `task_write_int`, `task_write_hex` |
| `kernel/semaphore.c` | `sem_write_uint`, `sem_write_int` |
| `kernel/delay_queue.c` | `delay_queue_write_uint`, `delay_queue_write_int` |
| `kernel/dispatcher.c` | `dispatcher_write_int` |
| `kernel/itron_api.c` | `itron_api_write_int`, `itron_api_write_uint32` |
| `kernel/preemption.c` | `preemption_irq_write_int` |
| `kernel/task_context.c` | `context_write_uint`, `context_write_int`, `context_write_hex` |
| `arch/x86_64/interrupt.c` | `arch_write_uint`, `arch_write_hex64` |
| `kernel/timer.c` | `timer_write_uint` |

同様に `task_state_to_string()` / `task_wait_reason_to_string()` も `task.c`, `delay_queue.c`, `task_context.c`, `preemption.c`, `itron_api.c` に分散している。

**修正方針:** `kernel/kernel_log.c` (または `hal/log.c`) に共通ログヘルパーを集約し、各モジュールからインクルードする。

---

### M-3: `sem_legacy_sig_sem_disabled()` がデッドコードになっている

**ファイル:** `kernel/semaphore.c:804-933`

この関数はソースコード内のどこからも呼び出されていない (grep 確認済み)。`sig_sem()` が `itron_api.c` に移行した際に残ったレガシーコードであり、名前に `_disabled` が付いているにもかかわらず `semaphore.h` から公開されている。

**修正方針:** 関数本体および semaphore.h からのプロトタイプ宣言を削除する。

---

### M-4: `task_mark_waiting_on_sem()` が READY → WAITING 遷移を許可している

**ファイル:** `kernel/task.c:1059`

```c
if (task->state != TASK_STATE_READY && task->state != TASK_STATE_RUNNING) {
    return TASK_ERR_BAD_STATE;
}
```

μITRON では `wai_sem()` は task 文脈 API であり、RUNNING 状態のタスクのみが呼べる。READY タスクが WAITING に落ちると、`current` でないタスクが WAITING 状態になり dispatch_pending の整合性が崩れる恐れがある。

**修正方針:** `TASK_STATE_RUNNING` のみを許可し、READY からの遷移を拒否する。

---

### M-5: `delay_queue.c` が task モジュールの内部フィールドを直接書き換えている

**ファイル:** `kernel/delay_queue.c:454`

```c
if (task != NULL) {
    task->delay_ticks_remaining = after;  // task モジュール外からの直接書き換え
}
```

`delay_ticks_remaining` は TCB のフィールドであり、その更新は task モジュールの API 経由であるべき。現状では delay_queue モジュールが task モジュールの内部状態を直接変更しており、状態所有権の原則に反する。

**修正方針:** `task_update_delay_remaining(task_id, remaining)` という API を task モジュールに追加し、delay_queue はこれを呼ぶ。

---

### M-6: `dispatch_pending` が IRQ 開始時に task context 由来の pending を消去する

**ファイル:** `kernel/preemption.c:213`

```c
const char *preemption_evaluate_from_irq(void) {
    dispatch_pending_clear_for_test_or_later_boundary();  // ← 毎 IRQ で無条件クリア
    current = dispatcher_get_current();
    ...
```

`sta_tsk()` や `sig_sem()` が設定した task-context 由来の dispatch pending が、次の timer IRQ の先頭でクリアされる。これらの pending は実際にはまだ消費されていないため、preemption 要求が失われる。

**修正方針:** IRQ 由来と task-context 由来の pending を別フラグで管理するか、IRQ 評価開始時には IRQ 由来 pending のみをクリアする設計にする。

## Minor — 改善推奨

### m-1: Doxygen 冗長/不整合箇所

**ファイル:** `kernel/task.c:548`

`task_dump()` の Doxygen コメントが `task_create_dormant()` の前に置かれている (line 548 の `/**` は `task_dump()` の説明から始まっているが、コードブロックは `task_create_dormant()` の実装)。実際の `task_dump()` 定義は line 636。これは Doxygen の `@brief` タグが実体と乖離している。

---

### m-2: `TASK_ID_MAX` / `SEM_ID_MAX` がマジックナンバー

**ファイル:** `kernel/task.c:27`, `kernel/semaphore.c:26`

```c
#define TASK_ID_MAX 2147483647  // INT_MAX と同値だが明示的でない
#define SEM_ID_MAX  2147483647  // 同上
```

`<limits.h>` の `INT_MAX` を使うか、定義を共通ヘッダに置くことで意図が明確になる。

---

### m-3: スケジューラが毎回 256 スロットをフルスキャン

**ファイル:** `kernel/scheduler.c:62-93`

`scheduler_select_next()` は `MAX_TASKS=256` の全スロットを毎回線形探索する。`scheduler_select_preemption_candidate()` はこれを内部で呼ぶため、1 tick あたり最大 2 回のフルスキャンになる。教育用途では問題ないが、タスク数が増えると性能劣化する。

**v2.0 への布石:** 優先度ビットマップまたは ordered ready queue の導入を検討。

---

### m-4: `sem_reset_wait_queue` が毎初期化時に 256 エントリをゼロクリア

**ファイル:** `kernel/semaphore.c:41-53`

```c
static void sem_reset_wait_queue(semaphore_t *sem) {
    for (index = 0; index < MAX_TASKS; index++) {  // MAX_TASKS=256 を毎回ゼロクリア
        sem->wait_queue[index] = 0;
    }
```

`MAX_TASKS` セマフォがあると 256×256=65536 回の書き込みが初期化時に発生する。実際の wait_count の最大値 (現実的には数タスク) に合わせた固定長への変更を検討。

---

### m-5: エラーコードが μITRON 4.0 仕様と異なる

**ファイル:** `kernel/include/itron_api.h`

```c
#define E_ID    (-1)    // μITRON 4.0: -18
#define E_PAR   (-2)    // μITRON 4.0: -17
#define E_CTX   (-3)    // μITRON 4.0: -25
#define E_OBJ   (-4)    // μITRON 4.0: -41
#define E_TMOUT (-5)    // μITRON 4.0: -50
```

将来の μITRON 準拠 API 移行時に全アプリコードの修正が必要になる。現時点でも `itron_error_name()` のような変換 API が存在するため、内部値を仕様値へ揃えておくと移行コストが低くなる。

## Suggestion — 将来的な改善案

### S-1: dispatch_pending の設計を複数要求対応に拡張する

**ファイル:** `kernel/dispatch_pending.c`

現状は `dispatch_pending_state` が単一の pending を保持する。Mutex/EventFlag/Mailbox を追加すると task context から複数の dispatch 要求が同時に生じる可能性があり、現設計ではキューイングできない。IRQ 由来 / task-start 由来 / wakeup 由来を分離して管理するか、pending キューを導入する設計変更を検討。

---

### S-2: ログ出力の一元化と将来の ringbuffer 化

複数モジュールに分散した `hal_console_write` 呼び出しを `kernel_log(level, fmt, ...)` のような薄い抽象に置き換えると、将来の ringbuffer ログや非同期出力への移行が容易になる。特に IRQ コンテキストでのシリアル出力は polling のため性能問題になり得る。

---

### S-3: `task_context.c` の smoke 用グローバル変数を分離する

**ファイル:** `kernel/task_context.c:26-34`

```c
static task_context_t boot_context;
static tcb_t *active_task;
static tcb_t *task_to_task_smoke_from;
static tcb_t *task_to_task_smoke_to;
```

`task_to_task_smoke_from/to` は boot-time smoke テスト専用の変数であり、production の dispatcher 経路には使われるべきでない。これらを `#ifdef SMOKE_TEST` または別ファイルに隔離することで、`task_context.c` の責務を「task context の保存・復元」に限定できる。

---

### S-4: rel_wai / sus_tsk に向けた wait_reason 拡張設計

**ファイル:** `kernel/include/task.h` (task_wait_reason_t)

`rel_wai()` を実装するには、任意の WAITING タスクを理由に関わらず READY に戻す必要がある。現状の `wait_reason` 別 `task_wake_*()` 関数群の設計では、`rel_wai()` に対応する汎用的な起床 API がない。`TASK_WAIT_REASON_ANY` を処理する `task_force_wake(id, E_RLWAI)` のような戻り値付き起床 API の検討が必要。

`sus_tsk()` は新しい状態 `TASK_STATE_SUSPENDED` またはフラグ (`tcb_t.is_suspended`) を必要とする。状態遷移図の拡張が必要になるため、v2.0 の早い段階で設計を確定させる。

## v2.0 へ向けた提案

### 1. コンテキストスイッチの完成

v1.0 の最大の未完成事項は「ブロッキング待ちタスクが待ち呼び出しの戻り先から再開できない」ことである。

v2.0 では以下を実装する:

1. **`task_mark_waiting_on_*()` 前に現在の実行コンテキストを TCB へ保存**: `rip` (戻りアドレス) を含む完全な callee-saved context の保存
2. **dispatcher_switch_to() の WAITING パス修正**: `boot_context` ではなく `from->context` を保存し、他タスクへ切り替える
3. **wakeup 後の context 復元**: WAITING タスクが READY → RUNNING になった時に、保存されたコンテキストから再開する

### 2. IRQ セーフなクリティカルセクション

`hal_interrupt.h` に `hal_interrupt_disable()` / `hal_interrupt_enable()` を追加し、task context API の入出口でこれを呼ぶ。または `dispatch_pending` を IRQ/task-context で書き込む際に原子操作を使う設計を検討する。

### 3. Ready Queue の導入

`scheduler_select_next()` の O(N) スキャンを廃止し、優先度ビットマップ (32 優先度 × bitmask) または doubly-linked ready queue に切り替える。`task.c` のテーブル管理と `scheduler.c` の選択責務を維持しつつ、テーブルの走査ではなくキューのデキューで選択できる設計へ移行する。

### 4. dispatch_pending の複数要求対応

IRQ 由来と task-context 由来の pending を分離し、どちらかが残っていれば次の消費境界で処理されるよう設計を変更する。Mutex/EventFlag/Mailbox 追加時に同時複数 wakeup が必要になるため、early に設計しておく価値がある。

### 5. dly_tsk の sleep_tsk 統合とタイムアウト統一

現状 `delay_queue` と semaphore の `wait_queue` が別管理になっており、`twai_sem()` で両方への登録が必要という複雑さがある。v2.0 では「全ての WAITING 状態は delay_queue エントリを持つ (timeout=無限は特別値)」という統一モデルへの移行を検討する。`sig_sem()` でのタイムアウト待ちタスク起床時の delay_queue 削除処理が不要になり、`sem_remove_waiter()` 相当の複雑な処理を削減できる。

## Zenn 記事用サマリー

### なぜレビューを実施したのか

本プロジェクトは教育目的の μITRON 風 RTOS を x86_64/QEMU 上で C 言語と Clang を使って実装してきた。第1章から第15章まで段階的に機能を追加し、タスク管理・優先度スケジューラ・セマフォ・遅延待ち・コンテキストスイッチ・timer IRQ などを実装した v1.0 として一区切りを迎えた。この節目に、設計上の問題点を洗い出し、v2.0 の実装方針を明確にするためにアーキテクチャレビューを実施した。

### どのような観点でレビューしたのか

(1) **コード品質**: 重複コード・命名規則・Doxygen コメントの整合性を確認した。特に数値出力ヘルパーが 9 ファイルに分散していることや、`task_configure_slot()` が追加されたにもかかわらず `task_register()` が使っていない点を指摘した。

(2) **RTOSアーキテクチャと責務分離**: scheduler が「選択のみ」であること、dispatcher が「current 管理のみ」であるべきことを軸に、各モジュールが互いの所有権を侵犯していないか検証した。

(3) **状態遷移の正確性**: DORMANT/READY/RUNNING/WAITING の遷移を全関数で追跡し、不正遷移パスを特定した。

(4) **IRQ 安全性**: timer IRQ ハンドラと task context API の並行実行時のレース条件を分析した。

(5) **μITRON 適合性と拡張性**: 将来の Mutex/EventFlag/Mailbox 実装に向けた設計上の障壁を確認した。

### どのような問題が見つかったのか

最も重大な問題は「ブロッキング待ちタスクの実行コンテキストが保存されない」ことだ。`wai_sem()` や `slp_tsk()` でタスクが WAITING になって別タスクへ切り替わる際、`dispatcher.c` は WAITING タスクのコンテキストを保存せずに `boot_context` から次タスクへ進む設計になっている。このため、WAITING タスクが wakeup されて再 dispatch された場合、`wai_sem()` の呼び出し元に戻れず entry() 先頭から再実行されてしまう。v1.0 の smoke テストではタスクが1回しか実行されないため問題が顕在化していないが、本格的なブロッキング待ち/再開サイクルの実現には抜本的な修正が必要である。

次に、`dispatcher.c:268,273` で TCB の state フィールドを直接書き換えている点が判明した。task モジュールには `task_mark_running()` / `task_mark_ready_from_running()` という専用 API が存在するにもかかわらず、dispatcher はこれらを使わずポインタ経由で直接変更している。また、task context API (wai_sem/sig_sem/sta_tsk 等) に割り込み保護がなく、timer IRQ が発火すると `dispatch_pending` が消去されるレース条件が存在することも確認した。

コード品質面では、`write_uint/write_int/write_hex` の同一実装が 9 ファイルに分散していること、`sem_legacy_sig_sem_disabled()` が未使用のまま公開 API として残っていることなどが確認された。

### 今後どう活用するか

本レビューの結果を Priority 1 〜 3 の修正リストとして整理した。v2.0 では、コンテキストスイッチの完成 (ブロッキング/再開の正確な実装)、IRQ セーフなクリティカルセクションの導入、ready queue の設計を優先的に取り組む。また dispatch_pending の複数要求対応は Mutex/EventFlag 追加の前提となるため、v2.0 の早期に着手する。教育用 RTOS として「動作の観測しやすさ」を保ちつつ、本物の RTOS に必要な正確性を段階的に追加していくアプローチは引き続き維持する。
