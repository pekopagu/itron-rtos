## 1. M-4対応確認

**判定: OK**

### 根拠

**実装 (`kernel/task.c:1061`):**
```c
if (task->state != TASK_STATE_RUNNING) {
    return TASK_ERR_BAD_STATE;
}
```
`TASK_STATE_READY` の条件が削除され、RUNNING 一択になっている。

**Doxygen整合 (`kernel/task.c:1028-1033`, `kernel/include/task.h:362-370`):**
- task.c: 「第12章以降は task 文脈 API から呼ばれるため、対象は実行中の RUNNING task に限定する」に更新。
- task.h: 「第12章以降は `wai_sem()` の task 文脈 API 経路で使うため、対象 task は RUNNING である必要がある」に更新。
- 実装と一致している。

**smoke 調整 (`kernel/kernel.c:1396-1398`):**
```c
if (sem_wait_id > 0 &&
    dispatcher_commit_current(task_get_by_id(sem_wait_id)) == DISPATCHER_OK &&
    task_mark_waiting_on_sem(sem_wait_id, 99) == 0) {
```
READY の task を current (RUNNING) にしてから `task_mark_waiting_on_sem` を呼ぶ形に変更。`task_mark_waiting_on_delay` / `task_mark_waiting_on_sem_timeout` の既存パターンと一致する。

**ログ確認 (`docs/logs/qemu-serial.log:524-527`):**
```
[task] waiting: id=25 name=task_sleep_sem_wait wait_sem_id=99 wait_reason=semaphore state=WAITING
[wup-tsk] called: tskid=25 current id=25 ... state=WAITING
[wup-tsk] invalid state: tskid=25 state=WAITING reason=semaphore expected=WAITING reason=sleep
[wup-tsk] completed: result=E_OBJ action=invalid-state
```
WAITING 遷移が成功し、`wup_tsk` が sem-waiting タスクへの適用を E_OBJ で正しく拒否している。

### 気になる点

**smoke 内の current_task の一時的な矛盾 (許容範囲)**

`dispatcher_commit_current(sem_wait_id)` 成功後、`task_mark_waiting_on_sem(sem_wait_id, 99)` が WAITING に変える。この時点で `current_task` は WAITING タスクを指す。その後 `wup_tsk` が E_OBJ を返し、次の `dispatcher_commit_current(delay_wait_id)` が `current_task` を上書きするまで current_task が WAITING タスクを指したままになる。これは boot-time smoke の「手動で状態を組み立てる」パターンと一致しており、production の wai_sem() 経路では起きないため許容範囲。

## 2. C-1対応確認

**判定: OK、ただし1件の気になる点あり**

### 根拠

**直接書き換えの排除 (`kernel/dispatcher.c:266-294`):**
```c
// 修正前
from->state = TASK_STATE_READY;   // 直接書き換え
to->state = TASK_STATE_RUNNING;   // 直接書き換え

// 修正後
result = task_mark_ready_from_running(from->id);  // API 経由
...
result = task_mark_running(to->id);               // API 経由
...
current_task = to;  // task_mark_running() 成功後にのみ更新
```

**RUNNING→READY が API 経由 (`task.c:882`):**
`task_mark_ready_from_running()` は RUNNING のみを許可し、wait メタデータをクリアする。

**READY→RUNNING が API 経由 (`task.c:817-833`):**
`task_mark_running()` は READY のみを許可し、wait_reason / wait_sem_id / delay_ticks_remaining をクリアする。

**`current_task` が `task_mark_running()` 成功後のみ更新:**
`dispatcher.c:285-295` において、エラー時は return を返して `current_task = to` (line 295) へ到達しない。

**WAITING タスクを誤って READY 化しない:**
line 266 の `from->state == TASK_STATE_RUNNING` ガードにより、WAITING の from は `task_mark_ready_from_running()` をスキップする。

**ログ確認 (`docs/logs/qemu-serial.log`):**
```
line 57:  [dispatcher] state transition: from id=2 name=task_b RUNNING->READY
line 58:  [dispatcher] state transition: to id=3 name=task_c READY->RUNNING
line 172: [dispatcher] state transition: to id=1 name=task_a READY->RUNNING   (from=6はWAITINGなのでREADY化ログなし)
line 475: [dispatcher] state transition: to id=23 name=task_sleep_next READY->RUNNING  (from=22はWAITINGなのでREADY化ログなし)
```
WAITING からの切り替え時に from 側の `RUNNING->READY` ログが出ないことも正しい動作。

### 気になる点

**部分更新リスク (`dispatcher.c:270-293`): 理論的にのみ到達**

```c
result = task_mark_ready_from_running(from->id);   // 成功: from → READY
// ...
result = task_mark_running(to->id);                // 仮に失敗
return DISPATCHER_ERR_BAD_STATE;                   // ← 返るが from は既に READY
```

`task_mark_ready_from_running` 成功後に `task_mark_running` が失敗すると、`from` が READY になって RUNNING タスクがいない中途半端な状態になる。ただし:
- `to` は入口 (line 245) で READY を事前検証済み
- v1.0 の single-threaded boot context では IRQ がここで割り込まない
- よって実際には到達しない

v2.0 でマルチタスク文脈が本格化する際に、ロールバック機構または前検証の強化が必要になる。

## 3. C-3 / M-6対応確認

**判定: OK、新リスク1件あり**

### 根拠

**source enum の追加 (`kernel/dispatch_pending.c:23-29`):**
```c
typedef enum {
    DISPATCH_PENDING_SOURCE_NONE = 0,
    DISPATCH_PENDING_SOURCE_IRQ,
    DISPATCH_PENDING_SOURCE_TASK
} dispatch_pending_source_t;
```
`dispatch_pending_state_t` に `source` フィールドが追加された。

**preemption.c の変更 (`kernel/preemption.c:213`):**
```c
// 修正前
dispatch_pending_clear_for_test_or_later_boundary();   // 全クリア

// 修正後
dispatch_pending_clear_irq_for_evaluation();            // IRQ由来のみクリア
```

**`dispatch_pending_clear_irq_for_evaluation()` の実装 (`kernel/dispatch_pending.c:337-342`):**
```c
void dispatch_pending_clear_irq_for_evaluation(void)
{
    if (dispatch_pending_state.source == DISPATCH_PENDING_SOURCE_IRQ) {
        dispatch_pending_clear_with_reason(NULL);
    }
}
```
TASK source の pending は保持される。正しい実装。

**`dispatch_request_from_irq()` の task-context pending 保護 (`kernel/dispatch_pending.c:200-208`):**
```c
if (dispatch_pending_state.requested &&
    dispatch_pending_state.source == DISPATCH_PENDING_SOURCE_TASK) {
    return;   // IRQ request を上書きしない
}
```
未消費の task-context pending がある間は IRQ request がドロップされる。

**source の設定が一貫している:**
- `dispatch_request_from_irq()`: `source = DISPATCH_PENDING_SOURCE_IRQ` ✓
- `dispatch_request_from_task_start()`: `source = DISPATCH_PENDING_SOURCE_TASK` ✓
- `dispatch_request_from_task_wakeup()`: `source = DISPATCH_PENDING_SOURCE_TASK` ✓
- `dispatch_pending_clear_with_reason()`: `source = DISPATCH_PENDING_SOURCE_NONE` ✓

**全 clear の呼び出し箇所が適切に維持されている:**
- `dispatch_request_from_task_start()` / `_wakeup()` の NULL ガード: `dispatch_pending_clear_for_test_or_later_boundary()` (全クリア) を維持。task-context API 呼び出しが無効引数だった場合に古い pending を残さない防御的処理として適切。
- smoke 初期化 (`kernel.c:1233`, `1317`): `dispatch_pending_clear_for_test_or_later_boundary()` を維持 ✓

### 気になる点: stale task-context pending による IRQ pending サイレントドロップ (新リスク)

**ファイル:** `kernel/dispatch_pending.c:200-208`

`sig_sem()` が tick-ready-smoke 前のフェーズ (sem-smoke) で `dispatch_request_from_task_wakeup()` を呼び、task-context pending を設定した。この pending は dly-smoke / twai-smoke / tick-ready-smoke にまたがって消費も削除もされなかった (`dispatch_pending_clear_for_test_or_later_boundary()` が tick-ready-smoke 前に呼ばれないため)。

tick-ready-smoke の timer IRQ 評価 (log line 346-349) で:
1. `preemption_evaluate_from_irq()` が `dispatch_pending_clear_irq_for_evaluation()` を呼ぶ → source=TASK なのでスキップ
2. `dispatch_request_from_irq(current=14, candidate=13)` を呼ぶ → source=TASK の pending があるため **IRQ request がサイレントにドロップ**
3. log は `[preempt-irq] decision: result=request-switch` (line 348) と出力するが、実際の pending はフェーズをまたいで残った古い task-wakeup (from=7, to=6) が残存

```
line 348: [preempt-irq] decision evaluated: result=request-switch reason=higher-priority-ready
line 349: [dispatch-pending] requested: reason=task-wakeup from id=7 name=task_wai_sem_to to id=6 name=task_wai_sem_from
```

`preempt-irq` ログが「切替要求を発した」と言いつつ、`dispatch-pending` ログには sem-smoke 由来の古い task-wakeup pending が見えるという観察上の矛盾が生じる。

**v1.0 の影響:** smoke は「観測のみ、dispatch 実行なし」のため動作上の問題はない。また修正計画のコメントに「単一pendingのv1.0モデルでは」と明示されており、設計上の既知制約。

**v2.0 への影響:** 実際に dispatch を実行するようになった場合、未処理 task-context pending が残存している tick で IRQ preemption が 1 tick 遅延する可能性がある。長期間残存する task-context pending は IRQ を繰り返しブロックし続けるリスクがある。S-1 (dispatch_pending 複数対応) が前提対策になる。

## 5. 今回の修正で新たに発生した懸念

### 懸念 A: stale task-context pending による IRQ evaluation サイレントドロップ (前述)

**ファイル/関数:** `kernel/dispatch_pending.c:200-208` (`dispatch_request_from_irq`)

- **現象:** sig_sem() などが設定した task-context pending が、その pending を消費する dispatch 境界に到達する前に timer IRQ が複数回発火すると、各 IRQ の preemption 評価結果が pending に反映されない。
- **v1.0 の実害:** なし。観測モデルのため dispatch は実行されない。
- **v2.0 の懸念:** 実際の dispatch を行う際に最大 1 tick の preemption 遅延。pending が長期間未消費だと複数 tick 遅延する。
- **対策:** S-1 (dispatch_pending 複数対応) で IRQ pending と task-context pending を独立に保持する。

### 懸念 B: smoke フェーズ間での dispatch_pending の持続

**ファイル:** `kernel/kernel.c` (smoke 関数の入口)

dly-smoke / twai-smoke / tick-ready-smoke の先頭に `dispatch_pending_clear_for_test_or_later_boundary()` がない。create-start-smoke と sleep-wakeup-smoke の先頭にのみ存在する。

今回の C-3/M-6 fix により、フェーズをまたいで残存する task-context pending が IRQ evaluation をブロックする効果が顕在化した。ログの読みやすさの観点から、各 smoke の先頭に pending clear を追加する検討を推奨する。ただし既存 smoke のログを変えるため、今回は修正しない。

### 懸念 C: C-1 fix の理論的な部分更新 (再掲)

**ファイル/関数:** `kernel/dispatcher.c:270-293` (`dispatcher_switch_to`)

`task_mark_ready_from_running(from)` 成功後に `task_mark_running(to)` が失敗すると RUNNING タスクなし状態が生じる。v1.0 では実際には到達しないが、v2.0 のマルチタスク環境では修正が必要。

## 7. 次に対応すべきこと

### Priority 1: C-2 対応 (最優先)

**問題:** ブロッキング待ちタスクの実行コンテキストが保存されていない。  
**ファイル:** `kernel/dispatcher.c:297-310`, `kernel/task_context.c`  
**内容:** `wai_sem()` でタスクが WAITING になって次タスクへ切り替わる際、WAITING タスクの CPU コンテキストが boot_context から切り替えられる。WAITING タスクが wakeup されると entry() 先頭から再実行される。  
**作業範囲:** `arch_context_switch` の from 保存、WAITING 入口での context 保存、wakeup 後の context 復元。変更範囲が大きいため、別単位として設計してから実装する。

### Priority 2: 懸念 A の抜本対応 (S-1)

**問題:** single pending model で IRQ pending と task-context pending が競合する。  
**ファイル:** `kernel/dispatch_pending.c`, `kernel/include/dispatch_pending.h`  
**内容:** IRQ 由来 pending と task-context 由来 pending を別フィールドで保持し、どちらも独立に設定・消費できる構造に変える。C-2 完了後に実施することで、実際の dispatch 実行時の preemption 遅延問題を解消できる。

### Priority 3: 懸念 B の smoke 整理

**問題:** smoke フェーズ間に dispatch_pending の明示的な清算がない。  
**ファイル:** `kernel/kernel.c` (dly-smoke / twai-smoke / tick-ready-smoke 入口)  
**内容:** 各 smoke 先頭に `dispatch_pending_clear_for_test_or_later_boundary()` を追加し、フェーズをまたぐ pending の持続を防ぐ。ログの読みやすさが向上する。
